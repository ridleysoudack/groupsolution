What We Learned

Exploring solutions to engage residents in the management of an expanding urban forest in Beaumont, Alberta.

ArcGIS Dashboards

The first solution Team.13 explored was ArcGIS Dashboards. Although initially promising, the team eventually decided to pivot to the more powerful ArcGIS Experience Builder.

City of Beaumont Tree Inventory Dashboard.

ArcGIS Dashboards offer a quick and simple way to visualize data in an interactive way. The core of the dashboard is the map which displays the data. For our project, we needed a solution that would display the current tree inventory of the City of Beaumont and provide some information about what is being displayed on the map.

The fastest way to achieve this was to create a simple map in ArcGIS Pro that contained the tree feature class. The map was then published to ArcGIS Online as a web map that is used as the main map in the Dashboard. The total number of trees, species, and amount of each species in the current extent is displayed and updates as the user zooms and pans around the city.

Dashboards are quick and simple but also lack advanced functionality. For example, the user does not have a way of filtering or only selecting a certain species of tree. Charting of many different species (as in the case of Beaumont) becomes challenging to display effectively on one chart. A cap on the maximum number of categories to be displayed on the chart had to be implemented or it would become too crowded and not legible. In the end, we decided that ArcGIS Dashboards lacked the amount of functionality that would be needed in our project scope.

Key Takeaways

  • Dashboards are simple and quickly display data.
  • Lack the functionality of other products such as Experience Builder
  • Primarily for viewing data.

ArcGIS Experience Builder

ArcGIS Experience Builder allows the user to create powerful web applications using data from ArcGIS Online. Through connecting to the web map and using the default widgets, the product is a versatile web application that allows the user to interact with the data for their own intended use. The tabs below highlight features from our learnings. Our solution was developed for desktop web browsers, but we plan on exploring additional mobile functionality in future.

From our early iterations it was clear that Experience Builder was going to be used in the final solution. Through its widgets, it presented more advanced functionality than ArcGIS Dashboards, while the simple Builder user interface meant interactions between the map and default widgets was simple to implement. At this early exploratory phase, the team worked on a number of iterations simultaneously, exploring different aspects of Experience Builder functionality, including chart widgets, multiple views, and survey integration. These early iterations featured widgets with largely default settings, which were not modified in any way to allow the user to filter by tree species or change the chart.

Key Takeaways
  • Easy method of customizing interaction with a web map
  • A lot of Esri's documentation for Experience Builder is not too helpful with niche applications of the default widgets

Added features include a query widget that allows the user to select any of the species in the tree inventory, as well as a height and diameter, to filter the added donut chart and the text on the view. The query can be limited by the current map extent or a drawn graphic. The chart filters based on the map extent, so if a certain species was selected in the query but was not present in the current extent, it would not appear on the chart.

An action was set up on the map frame to filter neighborhoods, parks, and trees based on the extent of the map. Then two separate list widgets were added and connected to the map frame, thereby filtering the lists to only show the parks/ neighborhoods within the current extent. Text connected to the map features was also added to calculate canopy cover, number of trees, as well as the park and neighborhood in view.

The survey for this view was set up to automatically take the AssetID of the selected tree on the map. As well as a section to add comments on the reason for the service request.

Through our contact at Beaumont the team obtained a more current tree inventory and the city's base data (neighborhoods, parks, trails, lakes, streets and city boundary). With updated data, the team combined learnings from previous iterations to build a new iteration featuring views, basic survey implementation, and even a basic calculation for canopy cover using area values for neighbourhoods polygons and summing the canopy area of the trees within.

It was during this iteration that the full functionality of Experience Builder, with relation to our proposed solution, was realized. The web application was sectioned into the three views (Tree Inventory, Neighborhoods/Parks, and the Service Request.

However, this iteration was also limited in that the calculation of canopy cover in the Neighborhoods/ Parks view was not always accurate. It was believed that acquiring the area of the map extent would require a customized widget, and as such the calculation of canopy cover at this stage used the area of the neighborhood. This makes the calculation accurate if only one neighborhood is in the extent, otherwise it will always take the area of the first neighborhood in the neighbourhood list widget, which will not represent screen extent.

Key Takeaways
  • With the updated data from our client, this iteration is much closer to the final solution with the three views.
  • Calculating canopy cover is limited to the area of the first neighborhood in the list.

While custom-coding a widget to address the issues regarding calculating canopy cover for the map extent was beyond the scope of this project, it was suggested to the team that they might be able to more easily make this calculation by applying a raster to the map and calculating the sum area of cells within the current map extent, to compare with the sum canopy area of all trees within the extent. The team explored this option, but was unfortunately only able to publish rasters as tile layers, which cannot be queried for analysis. Instead, the team came up with the solution of using the Create Fishnet geoprocessing tool in ArcGIS Pro to generate a vector grid that would serve the same base purpose.

Taking this approach, Team.13 created a transparent grid vector layer in ArcGIS Pro, with each cell sized at 30m x 30m. While not providing the exact area of the current extent (due to this resolution), the team found the sum of the area of each cell in view to be accurate enough for the simple percentage cover calculation being performed. The team did note that the addition of this large vector layer caused some stuttering in map zooming and panning, but not enough to significantly disrupt user experience.

The fourth iteration is our final solution with the exception of fine tuning the appearance of the page to match Beaumont's theme on their web pages.

Key Takeaways
  • Taking full advantage of Experience Builder, for our solution, required workarounds that could not be found anywhere using Esri's supplied documentation.
  • This iteration has the complete data, and all the widgets are set up for the final solution.

Experience Builder for Developers

The team opted to look into the locally installed version of Experience Builder to investigate possible solutions for obtaining the area of the current map extent.

Esri's supplied documentation for Experience Builder for Developers was enough to get it to run locally. The key is to run the Node.js Command Prompt, not the default windows one. Once npm in installed on the client and server folders https://localhost:3001 directs the user to their locally deployed Experience Builder.

Here the team ran into additional roadblocks, we were unable to import the beta version of our Experience Builder. Additionally, the team tried to save the previous experience builder as a template and then import that, however this failed as template was “built with a higher version of ArcGIS Experience Builder…”

To continue with this exact template reinstalling Experience Builder for Developers seemed to be the only path. However a simpler solution was discovered; to acquire the area of the current extent instead of customizing a widget we created a grid vector layer (see Iteration 4).

Key Takeaways
  • Allows for customized widgets that tailor the experience for the desired purpose
  • Due to a new solution for extent area, this was deemed unnecessary for our solution

The survey implemented with finalized symbology in the Experience. The Survey Widget has an option to not use a survey's assigned style, but unfortunately this option does not match the survey styling to the Experience styling.

This image shows the survey within Survey123. The team noted that certain question types appear slightly diferent in this view, as opposed to within the Survey Widget.

Implementation of Survey123 was a stretch goal for the team, as initially this integration seemed fairly complex. However, a “Survey” widget was discovered within Experience Builder that could seamlessly link a published ArcGIS Survey into the experience. Furthermore, the team discovered that the survey widget can be linked directly to map widgets within the experience, and a data connection can be established between the survey and a particular layer. When this connection is set, if the user selects a feature in the given layer within the map, questions in the survey will be auto-populated with data on that feature. Furthermore, by linking a “Location” question to a feature layer's shape field, the feature's spatial information can be copied over directly to the survey's results feature layer. By combining this data linking functionality with hidden questions in the constructed survey, it is possible to populate a large amount of feature attribute information to the survey when the user selects a feature, without cluttering the user interface.

This also suggests an exciting possibility, for future extensions of Team.13's work: because the Survey can receive location information, a live map of all submitted service requests can be integrated into a separate ArcGIS Experience view, for use by City of Beaumont internal staff. While not required for this particular implementation, this is something the team plans to incorporate in the larger implementation of the Collaborative Project.

Key Takeaways

  • The Experience Builder Survey Widget makes integrating Survey123 surveys a simple process.
  • The Survey Widget has a slightly modified appearance compared to viewing the survey in Survey123.

3D Interactive Scene

The Story Map can be configured as a continuously scrolling web page that can logically tell the story of your topic. For our project, the City of Beaumont would like to use a Story Map as a means of communicating their Urban Forest Management Strategy (UFMS) to the public.

In addition to including facts and information from Beaumont's UFMS, we wanted to provide the public with an interactive way of visualizing the cities current tree inventory. To accomplish this, a 3D Web Scene was created in ArcGIS Scene Viewer that includes City base data, as well as the Tree Inventory.

To improve the visual appearance of the scene, the trees were styled appropriately to represent 3D trees with heights taken from the height field of the data. This gives a more realistic appearance, and the user can easily distinguish younger trees from more mature trees throughout the city. Our aim is to style the trees more realistically according to species as well including views that highlight specific parks that would benefit from additional tree coverage.

Key Takeaways

  • Story Maps can help communicate a topic in a visually appealing way.
  • Their power comes from map integration.
  • 3D Scenes provide additional user interactivity to help communicate the concept.

Experience and Story Map Integration

Once the Story Map and Experience Builder implementations were finalized, the final steps were to integrate them with one another.

Stylized Experience. The team was dissapointed to note that font could not be matched to Story Map font selection.

Initially, the Team hoped to accomplish integration by placing both solutions in HTML iframes, and use an HTML page as a wrapper with navigation tabs to allow the user to seamlessly switch between the Story Map and the Experience. However, when this solution was tested, it quickly became clear that iframes caused a number of issues, including sluggish and unpredictable Experience functionality. Occasionally, one or both of the implementations would fail to load at all. Pivoting away from this approach, the team elected to simply create hyperlinks within the Experience and Story Map, allowing users to navigate between the two.

One additional issue in integration the team noted was that Experience Builder has much more limited style settings than Story Maps. In particular, Experience Builder has a very limited number of fonts available, and is missing many of the fonts included in the Story Maps builder. This was disappointing, as the team hoped to make the Story Map and Experience Builder aesthetics as similar as possible. While the team strived to match colour schemes where possible for this implementation, the team did not that Experience Builder for Developers allows for much more customization of an Experience, including importing custom fonts. The team intends to explore this further for the Collaborative Project implementation.

Key Takeaways

  • Iframes are not an effective way to implement Experiences/Story Maps within a website.
  • Experience Builder has limited style options.
